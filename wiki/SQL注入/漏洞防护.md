# SQL注入 — 漏洞防护

本文针对"SQL注入"模块中涉及的各类漏洞，分别给出代码层面的修复方案和防护建议。

---

## 一、经典SQL注入防护

### 1. 使用参数化查询（预编译）

这是最根本的防护措施：

```java
// ❌ 漏洞代码：字符串拼接
String sql = "SELECT * FROM users WHERE username='" + username + "'";
Statement stmt = conn.createStatement();
ResultSet rs = stmt.executeQuery(sql);

// ✅ 安全代码：参数化查询
String sql = "SELECT * FROM users WHERE username = ?";
PreparedStatement pstmt = conn.prepareStatement(sql);
pstmt.setString(1, username);
ResultSet rs = pstmt.executeQuery();
```

### 2. 输入验证与白名单

对于无法参数化的场景（如表名、列名），使用白名单校验：

```java
// 排序字段白名单
private static final Set<String> ALLOWED_ORDER_COLUMNS = Set.of("id", "name", "age", "created_at");

public String validateOrderColumn(String input) {
    if (!ALLOWED_ORDER_COLUMNS.contains(input.toLowerCase())) {
        throw new IllegalArgumentException("Invalid order column");
    }
    return input;
}
```

## 二、MyBatis 注入防护

### 1. 使用 #{} 替代 ${}

```xml
<!-- ❌ 漏洞代码 -->
<select id="searchByName" resultType="User">
    SELECT * FROM users WHERE name = '${name}'
</select>

<!-- ✅ 安全代码 -->
<select id="searchByName" resultType="User">
    SELECT * FROM users WHERE name = #{name}
</select>
```

### 2. 特殊场景的处理方案

#### 动态表名

```xml
<!-- ❌ 漏洞 -->
<select id="searchLogs" resultType="Map">
    SELECT * FROM logs_${tableName}
</select>

<!-- ✅ 安全：使用白名单 + choose -->
<select id="searchLogs" resultType="Map">
    SELECT * FROM logs_${safeTableName}
</select>
<!-- Controller中做白名单校验 -->
```

#### ORDER BY

```xml
<!-- ❌ 漏洞 -->
ORDER BY ${orderColumn} ${orderDir}

<!-- ✅ 安全：白名单校验 -->
ORDER BY ${validatedOrderColumn} ${validatedOrderDir}
<!-- 在Java代码中对orderColumn和orderDir做白名单验证 -->
```

#### IN条件

```xml
<!-- ❌ 漏洞 -->
AND role IN (${roles})

<!-- ✅ 安全：使用 <foreach> -->
AND role IN
<foreach collection="roleList" item="role" open="(" separator="," close=")">
    #{role}
</foreach>
```

#### LIKE查询

```xml
<!-- ❌ 漏洞 -->
WHERE name LIKE '%${name}%'

<!-- ✅ 安全：使用 CONCAT + #{} -->
WHERE name LIKE CONCAT('%', #{name}, '%')
```

### 3. 全局配置拦截

```java
// MyBatis 插件：拦截所有 ${} 使用
@Intercepts({
    @Signature(type = Executor.class, method = "update", args = {MappedStatement.class, Object.class}),
    @Signature(type = Executor.class, method = "query", args = {MappedStatement.class, Object.class, RowBounds.class, ResultHandler.class})
})
public class SqlInjectionInterceptor implements Interceptor {
    @Override
    public Object intercept(Invocation invocation) throws Throwable {
        // 检查SQL中是否存在未参数化的用户输入
        MappedStatement ms = (MappedStatement) invocation.getArgs()[0];
        BoundSql boundSql = ms.getBoundSql(invocation.getArgs()[1]);
        String sql = boundSql.getSql();
        // 检测逻辑...
        return invocation.proceed();
    }
}
```

## 防护总结

| 漏洞类型 | 核心防护 | 关键代码改动 |
|---------|---------|-------------|
| 经典拼接SQL | 参数化查询（PreparedStatement） | 替换字符串拼接为 `?` 占位 |
| MyBatis `${}` 误用 | 替换为 `#{}` | `${name}` → `#{name}` |
| 动态表名/ORDER BY | 白名单校验 | 校验输入是否在允许列表中 |
| IN条件 | 使用 `<foreach>` | `${roles}` → `#{role}` 循环 |
| LIKE查询 | 使用 CONCAT + #{} | `'%${name}%'` → `CONCAT('%', #{name}, '%')` |
| 盲注 | 参数化 + 最小权限 | 使用 `#{}` + 限制数据库用户权限 |

**核心原则**：永远不要将用户输入直接拼接到SQL语句中，所有用户输入必须通过参数化查询传递。
