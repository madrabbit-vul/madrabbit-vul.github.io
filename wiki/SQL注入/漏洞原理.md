# SQL注入 — 漏洞原理

## 什么是SQL注入？

SQL注入（SQL Injection）是指攻击者将恶意SQL代码插入到应用程序的查询中，从而改变原有SQL语句的逻辑，实现对数据库的非授权操作。SQL注入长期占据 OWASP Top 10 的前列，是 Web 应用最常见、危害最大的漏洞之一。

SQL注入的核心问题：**应用程序将用户输入直接作为SQL语句的一部分执行，未进行充分的过滤或参数化处理。**

---

## 一、经典SQL注入（字符串拼接）

### 漏洞成因

当应用程序使用字符串拼接方式构造SQL语句时，用户输入的特殊字符（如单引号 `'`、分号 `;`、注释符 `--`）会被数据库解释为SQL语法的一部分，而非普通数据。

常见的注入类型：

- **登录绕过**：通过 `OR '1'='1'` 构造永真条件
- **UNION注入**：通过 `UNION SELECT` 合并查询获取其他表数据
- **盲注**：通过布尔条件或时间延迟逐字符提取数据

### 真实案例

2017年，某知名在线教育平台因登录接口未参数化处理，攻击者通过 `admin'--` 绕过密码验证登录管理员账户，导致数百万用户数据泄露。

### 风险等级

| 维度 | 评级 |
|------|------|
| 攻击难度 | 低（自动化工具成熟） |
| 影响范围 | 极高（可读取/修改/删除数据库所有数据） |
| 利用条件 | 需要一个存在拼接SQL的输入点 |

## 二、MyBatis框架SQL注入

### 漏洞成因

MyBatis 提供了两种参数占位符：

- `#{param}`：预编译参数化，安全，使用 `PreparedStatement` 的 `?` 占位符
- `${param}`：字符串直接替换，危险，相当于拼接SQL

当开发者误用 `${}` 时，即使在 MyBatis 框架中也会产生SQL注入漏洞。常见的误用场景包括：

1. **数字型注入**：`WHERE age = ${age}`（无需引号闭合）
2. **字符串型注入**：`WHERE name = '${name}'`（需要引号闭合）
3. **LIKE注入**：`WHERE name LIKE '%${name}%'`（闭合引号+百分号）
4. **动态表名注入**：`FROM logs_${tableName}`（表名不可参数化）
5. **ORDER BY注入**：`ORDER BY ${orderColumn}`（排序字段不可参数化）
6. **IN条件注入**：`AND role IN (${roles})`（IN列表不可参数化）
7. **盲注**：结合布尔/时间/报错条件逐字符提取数据

### 真实案例

2020年，多个使用 MyBatis 的企业系统被安全审计发现 `${}` 误用问题，攻击者通过 ORDER BY 注入结合 `IF()` 条件实现了数据窃取。

### 风险等级

| 维度 | 评级 |
|------|------|
| 攻击难度 | 中（需要识别 `${}` 注入点） |
| 影响范围 | 高（取决于SQL语句的权限） |
| 利用条件 | MyBatis Mapper 中使用了 `${}` 占位符 |
