# 跨站请求伪造(CSRF) — 漏洞原理

## 什么是CSRF？

跨站请求伪造（Cross-Site Request Forgery, CSRF）是指攻击者诱导已认证用户在不知情的情况下向目标网站发送恶意请求，利用用户的登录状态执行非授权操作。

CSRF的核心问题：**浏览器会自动携带目标域的Cookie/Session，攻击者利用这一点伪造请求。**

---

## 一、GET型CSRF

### 漏洞成因

当敏感操作（如修改邮箱、修改密码）通过GET请求完成时，攻击者可以通过图片标签、链接等方式触发请求：

```html
<img src="https://target.com/api/change-email?email=attacker@evil.com">
```

用户只要访问包含此图片的页面，浏览器就会自动发起GET请求，并携带用户的Cookie。

### 风险等级

| 维度 | 评级 |
|------|------|
| 攻击难度 | 低（一个img标签即可） |
| 影响范围 | 高（任何访问恶意页面的登录用户） |
| 利用条件 | 敏感操作使用GET方法 |

---

## 二、POST型CSRF

### 漏洞成因

即使使用POST方法，如果没有CSRF Token验证，攻击者仍然可以通过自动提交的隐藏表单发起攻击：

```html
<form action="https://target.com/api/change-password" method="POST">
    <input type="hidden" name="newPassword" value="hacked123">
</form>
<script>document.forms[0].submit();</script>
```

### 风险等级

| 维度 | 评级 |
|------|------|
| 攻击难度 | 中（需要构造表单） |
| 影响范围 | 高 |
| 利用条件 | 无CSRF Token验证 |

---

## 三、无效CSRF Token

### 漏洞成因

服务端虽然实现了CSRF Token机制，但只验证Token是否有效，不验证Token是否属于当前用户。攻击者可以使用自己账号获取的有效Token，构造针对受害者的CSRF请求。

```
正常验证：Token有效 && Token属于当前用户 ✅
漏洞验证：Token有效 ← 缺少归属校验 ❌
```

### 风险等级

| 维度 | 评级 |
|------|------|
| 攻击难度 | 高（需要理解Token机制） |
| 影响范围 | 高 |
| 利用条件 | 有CSRF Token但未绑定用户 |
