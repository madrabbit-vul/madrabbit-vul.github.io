# 认证与会话安全 — 漏洞原理

## 什么是认证与会话安全？

在 Web 应用中，**认证 (Authentication)** 是服务器确认"你是谁"的过程，**会话 (Session)** 是服务器在认证通过后颁发的一张"临时通行证"（通常是 Cookie 或 Token）。如果这两环节中的任何一个存在漏洞，攻击者就能冒充合法用户访问系统。

认证与会话安全漏洞的核心问题可以归结为一句话：**服务器在判断"你是谁"时犯了错。**

---

## 一、暴力破解 (Brute Force)

### 漏洞成因

当登录接口没有实施以下安全措施时，就存在暴力破解风险：

- **无频率限制 (Rate Limiting)**：允许同一 IP 在短时间内发起无限次登录请求
- **无验证码 (CAPTCHA)**：没有要求用户完成人机验证
- **无账户锁定**：连续失败后不会冻结账户
- **无 IP 封禁**：不会对异常 IP 采取限制

攻击者使用自动化脚本配合密码字典，可以在几分钟内尝试数千个密码组合。

### 真实案例

2019 年，某大型互联网公司因登录接口未做频率限制，攻击者使用 2 万条常见密码字典在 30 分钟内爆破出数千个弱密码账户。

### 风险等级

| 维度 | 评级 |
|------|------|
| 攻击难度 | 低（工具成熟，门槛极低） |
| 影响范围 | 高（所有弱密码用户均受影响） |
| 利用条件 | 仅需一个可访问的登录接口 |

## 二、任意密码重置 (Password Reset Bypass)

### 漏洞成因

密码重置流程通常包含三步：身份验证 → 验证码校验 → 重置密码。漏洞出现在**验证逻辑缺失**的地方：

1. **验证凭证与账号未绑定**：服务器验证了验证码/Token 的有效性，但没有校验该凭证是发给哪个账号的。攻击者用自己账号获取的凭证，去重置他人账号的密码。

2. **验证码可预测**：验证码使用简单算法生成，可被枚举或推算。

3. **响应信息泄露**：服务器在响应中返回了验证码或重置链接。

```
正常流程：
  用户A 请求验证码 → 验证码发给A → 用A的凭证重置A的密码 ✅

漏洞利用：
  攻击者B 请求验证码 → 获得B的凭证 → 用B的凭证重置A的密码 ❌（凭证未绑定用户）
```

### 真实案例

2017 年，某社交平台密码重置接口仅验证短信验证码是否正确，未校验验证码与手机号的绑定关系。攻击者用自己的手机号获取验证码后，将请求中的手机号替换为目标用户的手机号，即可重置他人密码。

### 风险等级

| 维度 | 评级 |
|------|------|
| 攻击难度 | 中（需要理解业务流程） |
| 影响范围 | 高（可重置任意用户密码） |
| 利用条件 | 需要一个已注册账号 |

## 三、JWT 弱密钥爆破 (JWT Weak Key Brute Force)

### 漏洞成因

JWT (JSON Web Token) 由三部分组成：`Header.Payload.Signature`。其中 Signature 使用密钥对前两部分进行签名，防止内容被篡改。

当使用 HS256 (HMAC-SHA256) 对称加密算法时，签名和验证使用**同一个密钥**。如果密钥强度不足（如使用常见单词、短字符串、默认密钥），攻击者可以：

1. 获取一个合法的 JWT Token
2. 使用字典或工具（如 hashcat、jwt-cracker）离线爆破签名密钥
3. 用爆破出的密钥伪造任意用户的 Token

```
JWT 结构（Base64 编码）：
  Header   : {"alg":"HS256","typ":"JWT"}
  Payload  : {"username":"admin","role":"ADMIN"}
  Signature: HMACSHA256(base64(Header) + "." + base64(Payload), secret_key)

如果 secret_key = "secret123"，攻击者可在数秒内爆破成功
```

### 密钥填充问题

HS256 要求密钥长度至少 256 位（32 字节）。有些开发者会将短密钥通过填充达到长度要求，例如重复拼接。这种填充方式并不会增加密钥的熵值，爆破短密钥后即可还原填充后的完整密钥。

### 真实案例

2015 年，某知名 SaaS 平台的 JWT 密钥为 `secret`，被安全研究员在数分钟内爆破成功，随后伪造了管理员 Token 获取了系统最高权限。

### 风险等级

| 维度 | 评级 |
|------|------|
| 攻击难度 | 中（需要爆破工具和密钥字典） |
| 影响范围 | 极高（可伪造任意用户身份） |
| 利用条件 | 需要获取一个合法 JWT Token |
